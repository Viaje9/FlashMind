## Context

目前牌組詳情頁的卡片列表已經有搜尋與單一篩選下拉，且過濾邏輯集中在 `apps/web/src/app/pages/deck-detail/deck-detail-filter.domain.ts` 的前端 domain 函式。卡片列表 API 已回傳每筆列表項目的 `direction`，並在牌組啟用反向學習時展開 `FORWARD` 與 `REVERSE` 兩種列表項目，因此新增卡面方向篩選不需要改動 Prisma 資料模型。

## Goals / Non-Goals

**Goals:**

- 在既有牌組卡片列表中新增卡面方向篩選，讓使用者可切換檢視正面卡片或反面卡片。
- 僅在牌組列表內實際存在反面卡片項目時顯示方向下拉，避免沒有作用的控制項。
- 維持既有搜尋、到期篩選與排序行為，讓方向篩選可以和其他條件一起運作。

**Non-Goals:**

- 本次不調整卡片列表 API 回傳格式，也不新增 server-side filter 參數。
- 本次不變更卡片列表排序規則、分頁策略或反向學習資料的產生方式。
- 本次不新增多選方向條件或更細的方向狀態統計。

## Decisions

1. 沿用前端本地過濾模式擴充方向條件

- 決策：在 `deck-detail-filter.domain.ts` 增加方向篩選參數，讓搜尋字串、既有 filter 與卡面方向條件在同一個 domain 函式中組合判斷。
- 理由：現有卡片列表資料已包含 `direction`，前端可直接完成需求，避免不必要的 API 契約變更與跨模組成本。
- 替代方案：改由 API 接收 `direction` 查詢參數可減少前端運算，但會擴大 OpenAPI、client 產生與後端測試範圍，本次不採用。

2. 以條件式 UI 顯示方向下拉

- 決策：頁面根據目前載入的卡片列表是否存在任一 `REVERSE` 項目，決定是否渲染方向下拉。
- 理由：需求明確指出當牌組只有正面卡片時不顯示該選項；直接以列表資料判斷最貼近實際可見內容，也不需額外請求 deck metadata。
- 替代方案：以 `deck.enableReverse` 判斷較簡單，但若牌組已開啟反向學習、列表當前仍只有正面項目，會顯示無效選單，不符合需求。

3. 方向篩選採單選且預設為全部卡面

- 決策：方向下拉提供 `全部卡面`、`正面卡片`、`反面卡片` 三種單選值，預設為全部卡面。
- 理由：需求是快速縮小到單一方向檢視；單選可保留既有操作心智模型，也能與現有 Filter overlay 並列而不增加複雜度。
- 替代方案：切換式 segmented control 視覺上更直接，但在手機版空間較緊，且需要額外處理和現有 filter 區塊的一致性，本次不採用。

4. 方向篩選與既有條件採交集

- 決策：方向篩選 SHALL 與搜尋字串、到期條件一起套用，只有同時符合的列表項目會顯示。
- 理由：使用者可能先找「某個字」再只看反面卡片，交集邏輯最符合現有搜尋與 filter 行為。
- 替代方案：方向切換覆蓋其他條件會使列表結果跳動較大，也較難預期。

## Risks / Trade-offs

- [風險] 以目前載入的卡片列表判斷是否顯示方向下拉，若未來引入分頁或 server-side loading，可能只看到部分資料而誤判 → [緩解] 本次維持全量載入；若未來改成分頁，需一併調整為後端回傳可用方向 metadata。
- [取捨] 前端過濾條件增加後，domain 測試與元件狀態會更複雜 → [緩解] 依 TDD 補上 domain 測試，分別覆蓋顯示條件、方向篩選與條件交集。
- [風險] 方向下拉與既有 Filter/Sort 區塊並排後，行動版版面可能擁擠 → [緩解] 沿用現有控制項樣式並在頁面整合測試確認手機寬度下仍可操作。
